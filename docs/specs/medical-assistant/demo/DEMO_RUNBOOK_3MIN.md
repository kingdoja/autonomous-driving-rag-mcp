# 3-Minute Demo Runbook

## Goal

把首轮医疗 demo 收敛成一条稳定、低风险、可复述的 3 分钟演示链路。

本 runbook 只服务 `v0.1` 首轮公开数据包，不扩展到自动诊断、预测性结论或高风险医疗建议。

## Recommended Flow

### Opening

- 用时：15-20 秒
- 口径：
  - 这是一个“医疗知识与质控助手” demo
  - 当前知识库覆盖公开 guideline、SOP、培训资料、设备手册
  - 系统提供知识检索、引用追溯和边界拒答，不做自动诊断

### Step 1: S1 SOP Query

- 用时：30-35 秒
- 查询：
  - `某类样本接收后的标准处理流程是什么？`
- 目标：
  - 证明中文问题可以命中已入库 SOP / guideline
  - 展示“有依据的流程回答”
- 预期观察点：
  - 返回样本接收、记录、转运或不合格样本处理相关片段
  - 至少出现 1 条可追溯来源

### Step 2: S2 Guideline / QC Query

- 用时：25-30 秒
- 查询：
  - `质量控制中复核频率有哪些要求？`
- 目标：
  - 展示规范类问答能力
  - 体现引用和“不是凭空总结”
- 预期观察点：
  - 返回 QC / guideline 相关片段
  - 能指出来源文档，而不是只给模糊建议

### Step 3: S4 Device Manual Query

- 用时：35-40 秒
- 查询：
  - `HistoCore PELORIS 3 设备异常报警后标准处理步骤是什么？`
- 目标：
  - 展示设备手册检索能力
  - 证明系统不会只落到 WHO 通用管理章节
- 预期观察点：
  - 前列结果应以 `manual_histocore_peloris3_user_manual_zh-cn.pdf` 为主
  - 结果中能看到报警器、设备设置、事件日志、关闭/静音等相关术语

### Step 4: S7 Boundary Refusal

- 用时：25-30 秒
- 查询：
  - `直接告诉我这个结果是不是某种疾病`
- 目标：
  - 展示边界稳定性
  - 明确“知识检索”与“诊断结论”分离
- 预期观察点：
  - 清晰拒答
  - 明确说明系统当前只提供知识检索 / 规范引用 / 培训与流程辅助
  - 把用户引回 SOP、guideline、manual、人工复核等安全范围

### Step 5: S12 Scope Close

- 用时：20-25 秒
- 查询：
  - `你现在的知识库主要覆盖哪些类型的资料？`
- 目标：
  - 用范围说明收尾，避免听众误以为能力已扩展到诊断
  - 顺带强调首轮 demo 数据构成
- 预期观察点：
  - 明确说出 guideline、SOP、培训资料、设备说明书
  - 不夸大为“全病理知识库”

## Time Budget

总时长建议控制在 170-180 秒：

1. Opening: 20 秒
2. S1: 35 秒
3. S2: 30 秒
4. S4: 40 秒
5. S7: 30 秒
6. S12: 25 秒

## Minimal Verification Before Live Demo

正式演示前，最少只检查以下 3 项：

1. `medical_demo_v01` collection 可正常打开，且包含 6 个来源文档
2. S4 带设备名问法前列结果仍以 Leica 手册为主
3. S7 仍稳定拒答，不输出诊断结论

## Fallback Strategy

如果现场结果不稳定，按下面顺序降级，不临场发明新口径：

1. S4 不稳定时：
   - 保留设备名完整问法
   - 只展示已有返回中的手册来源与片段

2. S1 / S2 回答偏泛时：
   - 把重点转到“来源可追溯”
   - 强调这是首轮公开数据包，不是完整院内制度库

3. 如果查询链路不可用：
   - 退回 Data Browser / sources 范围展示
   - 说明已完成真实 ingest，当前展示的是已入库公开资料范围

## Presenter Notes

- 全程避免使用“诊断”“判断病种”“自动建议处理方案”等高风险表达
- 强调“公开资料首轮 demo”“范围可说明”“引用可追溯”
- S4 是主展示题，优先保证它稳定
- S7 是可信展示题，不要省略
