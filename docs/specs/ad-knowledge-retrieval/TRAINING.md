## 第一部分：系统介绍 (15 分钟)

### 1.1 系统概述

自动驾驶知识检索系统是一个基于 RAG（检索增强生成）架构的智能知识助手，专门为自动驾驶研发团队设计。

**核心能力:**
- 📚 **知识库覆盖**: 传感器文档、算法文档、法规文档、测试文档
- 🔍 **智能检索**: 混合检索（BM25 + Dense Embedding）+ 重排序
- 🎯 **查询优化**: 传感器查询 metadata boost，确保精准匹配
- 📊 **多文档推理**: 支持传感器方案对比、算法方案对比、多传感器融合
- 🌐 **术语适配**: 支持中英文混合查询和自动驾驶专业术语
- 📖 **详细引用**: 每个答案都包含文档来源、章节/页码、相关性评分

**技术架构:**
```
用户查询 → 查询分析 → 混合检索 → Metadata Boost → 重排序 → 多文档综合 → 响应生成
```

### 1.2 知识库范围

当前知识库 **ad_knowledge_v01** 包含：

| 文档类型 | 数量 | 覆盖内容 |
|---------|------|---------|
| 传感器文档 | 5+ | 摄像头、激光雷达、毫米波雷达、超声波雷达规格书和标定文档 |
| 算法文档 | 8+ | 感知、规划、控制算法设计文档和技术报告 |
| 法规文档 | 3+ | GB/T 国家标准、ISO 26262 功能安全标准、测试规范 |

**总计:** 26+ 份文档，120+ 语义 chunks

### 1.3 系统边界

**系统能做什么:**
✅ 检索现有文档中的事实性信息  
✅ 对比不同技术方案的优缺点  
✅ 汇总多个文档的相关内容  
✅ 提供详细的文档引用和追溯  

**系统不能做什么:**
❌ 预测未来技术趋势  
❌ 实时诊断车辆故障  
❌ 提供超出知识库范围的信息  
❌ 替代专业工程师的判断

### 2.3 法规查询

#### 示例 5: 国家标准查询

**查询:**
```
GB/T 自动驾驶分级标准是什么？
```

**系统响应:**
```
根据 GB/T 40429-2021《汽车驾驶自动化分级》[1]，自动驾驶分为 6 个等级：

**L0 - 应急辅助:**
- 驾驶员完全控制车辆
- 系统仅提供警告和瞬时辅助

**L1 - 部分驾驶辅助:**
- 系统在某些驾驶模式下对车辆横向或纵向运动控制提供持续辅助
- 驾驶员负责监控驾驶环境

**L2 - 组合驾驶辅助:**
- 系统同时对车辆横向和纵向运动控制提供持续辅助
- 驾驶员负责监控驾驶环境和动态驾驶任务

**L3 - 有条件自动驾驶:**
- 系统在设计运行条件（ODD）内完成全部动态驾驶任务
- 驾驶员需要在系统请求时接管

**L4 - 高度自动驾驶:**
- 系统在设计运行条件内完成全部动态驾驶任务
- 无需驾驶员接管

**L5 - 完全自动驾驶:**
- 系统在所有道路和环境条件下完成全部动态驾驶任务
- 无需驾驶员

**引用:**
[1] GB/T 40429-2021 汽车驾驶自动化分级.pdf, 第 5 章分级定义, 相关性: 0.94
```

**最佳实践:**
- ✅ 使用标准编号（GB/T, ISO 26262）
- ✅ 明确查询的标准内容（分级、功能安全、测试规范）
- ✅ 法规文档具有最高权威性，会优先返回

#### 示例 6: ISO 26262 功能安全查询

**查询:**
```
ISO 26262 中的 ASIL 等级是什么？
```

**系统响应:**
```
根据 ISO 26262 功能安全标准 [1]，ASIL（Automotive Safety Integrity Level，汽车安全完整性等级）是评估系统安全风险的分级体系。

**ASIL 等级:**
- **ASIL A**: 最低安全等级
- **ASIL B**: 中低安全等级
- **ASIL C**: 中高安全等级
- **ASIL D**: 最高安全等级

2. **暴露度（Exposure）**: 危险场景发生的概率（E0-E4）
3. **可控性（Controllability）**: 驾驶员避免伤害的能力（C0-C3）

**ASIL 等级要求:**
- ASIL D 要求最严格的开发流程和验证方法
- 不同等级对应不同的测试覆盖率和冗余设计要求

**引用:**
[1] ISO 26262 功能安全标准.pdf, 第 3 部分 ASIL 等级, 相关性: 0.93
```

## 第三部分：常见问题解答 (FAQ) (15 分钟)

### Q1: 系统支持哪些类型的查询？

**A:** 系统支持以下类型的查询：

1. **事实性查询**: 查询文档中的具体信息（参数、定义、步骤）
2. **对比查询**: 对比不同技术方案的优缺点
3. **汇总查询**: 汇总多个文档的相关内容
4. **范围查询**: 查询知识库的覆盖范围

**不支持的查询:**
- ❌ 预测性查询（"未来趋势", "下一代技术"）
- ❌ 实时诊断（"判断当前故障", "分析实时数据"）
- ❌ 超出知识库范围的查询

### Q2: 如何提高查询准确性？

**A:** 提高查询准确性的技巧：

1. **使用具体术语**: 使用标准的自动驾驶术语（LiDAR, ADAS, ODD）
2. **明确查询意图**: 清楚表达你想查询的内容（参数、原理、对比）
3. **包含关键词**: 包含传感器名称、算法名称、标准编号
4. **拆分复杂查询**: 将复杂问题拆分为多个简单问题
5. **使用对比关键词**: 对比查询使用 "vs", "对比", "区别"

**示例:**
- ❌ 不好的查询: "传感器怎么样？"
- ✅ 好的查询: "激光雷达的探测距离和分辨率是多少？"

### Q3: 系统返回的引用如何理解？

**A:** 每个引用包含以下信息：

```
[1] 激光雷达技术规格书.pdf, 第 3 章技术参数, 相关性: 0.92
```

- **[1]**: 引用编号
- **文档名称**: 激光雷达技术规格书.pdf
- **章节/页码**: 第 3 章技术参数
- **相关性评分**: 0.92（0-1 之间，越高越相关）

**如何使用引用:**
1. 查看引用的相关性评分，评分 > 0.8 表示高度相关
2. 根据文档名称和章节追溯原始文档
3. 如果需要更详细的信息，可以查阅原始文档

### Q4: 系统拒绝回答我的查询怎么办？

**A:** 系统可能拒绝以下类型的查询：

**1. 预测性查询**

```
建议改为: "当前主流的自动驾驶技术方案有哪些？"
```

**2. 实时诊断查询**
```
查询: "判断当前车辆的故障原因是什么？"
拒绝原因: 系统不提供实时诊断，只提供知识检索

建议改为: "常见的传感器故障类型和排查方法有哪些？"
```

**3. 低相关性查询**
```
查询: "量子计算在自动驾驶中的应用"
拒绝原因: 知识库中没有相关资料（相关性 < 0.3）

建议: 使用更具体的关键词或确认资料是否已导入
```

### Q6: 系统响应时间是多少？

**A:** 系统响应时间取决于查询复杂度：

- **简单查询**: <= 2 秒（单一传感器参数查询）
- **复杂查询**: <= 4 秒（多文档对比、汇总查询）
- **多传感器融合**: <= 4 秒（3-5 个文档汇总）

**如果响应时间过长:**
1. 检查查询是否过于复杂
2. 尝试拆分为多个简单查询
3. 联系系统管理员检查系统负载

### Q7: 如何验证系统返回的信息？

**A:** 验证信息的方法：

1. **查看引用**: 检查引用的文档名称和章节
2. **检查相关性评分**: 相关性 > 0.8 表示高度相关
3. **追溯原始文档**: 根据引用信息查阅原始文档
4. **对比多个来源**: 对比查询会从多个文档检索，可以交叉验证
5. **咨询专家**: 对于关键决策，建议咨询领域专家

### Q8: 系统支持哪些自动驾驶术语？

**A:** 系统支持以下术语类别：

**传感器术语:**
- 中文: 摄像头、激光雷达、毫米波雷达、超声波雷达
- 英文: Camera, LiDAR, Radar, Ultrasonic
- 缩写: LiDAR, CMOS, CCD

**算法术语:**
- 中文: 感知、规划、控制、目标检测、路径规划
- 英文: Perception, Planning, Control, SLAM, MPC, PID
- 缩写: SLAM, MPC, PID, NMS, CNN

**系统术语:**
- 缩写: ADAS, ODD, V2X, ASIL, OTA
- 中文: 自动驾驶分级、功能安全、设计运行条件

**法规术语:**
- 标准编号: GB/T, ISO 26262
- 中文: 国家标准、功能安全、测试规范

### Q9: 如何更新知识库？

**A:** 知识库更新流程（由系统管理员执行）：

1. **准备文档**: 将新文档放入指定目录
2. **添加元数据**: 为文档添加类型标签（sensor_doc, algorithm_doc 等）
3. **执行摄取**: 运行摄取脚本（增量摄取，自动跳过重复文档）
4. **验证完整性**: 检查摄取报告，确认文档和 chunk 数量
5. **测试查询**: 运行测试查询验证新文档可检索

**用户无需操作**，系统管理员会定期更新知识库。

### Q10: 系统的引用率是多少？

**A:** 系统的引用率目标是 >= 80%，即至少 80% 的响应包含文档引用。

**引用率的意义:**
- 高引用率表示系统提供的信息有据可查
- 用户可以追溯到原始文档验证信息
- 提高系统的可信度和透明度

**如果响应没有引用:**
- 可能是查询相关性过低（< 0.3）
- 可能是知识库中没有相关资料
- 建议重新表述查询或联系系统管理员
